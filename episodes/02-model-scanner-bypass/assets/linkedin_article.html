I Hid Malware in an AI Model — Which Scanner Caught It?

An AI model file can pass a security scanner and still run your attacker’s code — and this year that stopped being theoretical. In December 2025, JFrog’s security team disclosed three critical zero-days in picklescan (CVE-2025-10155, -10156, and -10157, each rated CVSS 9.3) — the exact scanner Hugging Face runs on every model uploaded to the Hub. All three let a malicious model slip past as “safe” and execute code on load. They were fixed in picklescan 0.0.31, but the deeper lesson is that the “unscannable model file” is having a moment, and the tools we lean on are not all equally hard to fool.

So I built it in miniature: one malicious model file carrying a benign canary payload, plus four scanners — picklescan, ModelScan, ModelAudit, and fickling — tasked with catching it. Everything runs locally in Docker on an isolated network with no internet egress, the payload is a benign canary, and the whole setup is reproducible at a pinned commit.

Why the model file itself became an attack surface

Most machine-learning models ship as pickle files — Python’s built-in object-serialization format. PyTorch’s .bin and .pt files are pickle under the hood; so are countless .pkl artifacts. And a pickle is not a data format the way JSON is. It is effectively a tiny program: it contains opcodes, and one of them, REDUCE, calls an arbitrary callable with arbitrary arguments. When you torch.load() a model, that program runs — before the model makes a single prediction. Roughly 45% of the most popular models on Hugging Face still use pickle, so this isn’t a fringe concern; it’s the default distribution format for a huge slice of the ecosystem.

The evasions, in plain terms

You don’t need to read pickle bytecode to reason about the bypasses. Three patterns cover most of what’s used against scanners:

  • Blocklist evasion. picklescan works from a blocklist of known-dangerous callables. A blocklist only catches names it already knows — reach the dangerous call by a route the list never enumerated and the string never matches.
  • Renamed or broken archives. A scanner has to recognize and open a file before it can inspect it. CVE-2025-10155 used a renamed pickle; ReversingLabs’ nullifAI models used 7z instead of zip so the scanner couldn’t open the container at all.
  • Payload-then-break. Put the payload at the start of the stream and a broken opcode right after it. The payload executes first; then the scanner hits the error, reports a parse failure, and never mentions the malware it already walked past.

Most of these aren’t exotic exploits. They’re the natural failure modes of a scanner that reasons about names and file types rather than intent.

The test, kept fair

Everything runs in Docker on a single bridge network marked internal, so no container can reach the internet. The embedded payload does exactly one thing: send a request to an in-network canary sink. When the payload runs, the sink logs a hit — that hit is my ground truth that the code executed.

From one base file I generate four variants, each isolating a single evasion axis against a known-bad global (builtins.eval) so the easy case is a fair true-positive baseline every scanner should catch: a plain baseline, the same global delivered via the STACK_GLOBAL opcode, a pickle renamed to a .bin extension, and a broken-archive container. Each scanner inspects the exact same files.

What each scanner caught

The fair baseline holds: with a known-bad eval, every scanner catches the plain case and the STACK_GLOBAL opcode form. The real misses are structural. ModelScan flagged eval as CRITICAL whenever it actually opened a file — but it routes by extension and never scanned the .bin-renamed or broken-archive files at all. picklescan inspects regardless of extension, but mis-parsed the broken-archive container and reported nothing. Only ModelAudit (behavioral) and fickling (allowlist) caught all four, because neither depends on extension routing or a cleanly-parseable container.

Totals: ModelAudit 4/4, fickling 4/4, picklescan 3/4, ModelScan 2/4. And the ground truth matters here: loading each file in an isolated loader confirmed all four malicious variants actually execute code, while all three clean models stayed silent — so every “safe”/skipped verdict above is a real miss on a genuinely live file, not a false alarm.

Cost and noise

Zero false positives on the clean set for all four. On speed and footprint: picklescan is effectively instant (sub-millisecond, ~23 MB peak); fickling ~50 ms / ~34 MB; ModelScan ~90 ms / ~35 MB; ModelAudit ~0.45 s / ~119 MB — its deeper opcode and pattern analysis costs the most, a fine trade for a pre-production gate.

The verdict

Scored on a weighted rubric (detection 30%, signal 15%, performance 10%, and install/UX/docs/value 45%):

  • ModelAudit — 8.6. Widest and deepest; the one I’d reach for when I actually want to find things, with opcode-level analysis and CI-friendly SARIF output.
  • fickling — 8.4. Its allowlist is the strongest posture against tricks nobody has enumerated yet.
  • picklescan — 8.3. The fast, ubiquitous default — patch it to the latest release and never rely on a blocklist alone.
  • ModelScan — 7.3. Flags known-bad globals well, but confirm what it’s actually scanning — extension routing let two live files through unscanned.

For any model you didn’t build yourself, the honest answer is to layer them: a fast blocklist scan as a CI gate, a deeper opcode/allowlist scan before anything reaches production, and a hard rule to update relentlessly — this category’s threat model shifts every few weeks.

Reproduce it yourself

Every number above comes from the lab run, reproducible at a pinned commit: github.com/sandhipveera/seceval-lab (@465a22c). It’s all benign — the payload trips a canary on an isolated network, so nothing leaves your machine and you can see where your results diverge. If they differ from mine, that’s the point — tell me.
